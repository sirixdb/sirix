--- Observations for iter#04 attack vectors ---

1. **HOTTrieReader.prefetchPage starts a virtual-thread-per-sibling**
   (line 516 of HOTTrieReader.java):
   ```java
   private void prefetchPage(PageReference ref) {
     Thread.startVirtualThread(() -> {
       final Page loaded = storageEngineReader.loadHOTPage(ref);
       if (loaded != null) ref.setPage(loaded);
     });
   }
   ```
   At PREFETCH_WINDOW=16 and 20 parallel workers each navigating their
   own sub-trees, this fires ~320 concurrent virtual threads at steady
   state — all competing for the FileChannel read pool + heating up
   the CPU (but also a source of lock/futex wait in the wall profile).
   Every depth level cascades more prefetch requests.

   Potential iter#04 lever: **bound prefetch concurrency or switch to
   an async `io_uring` batched submit** so we don't spawn thousands of
   virtual threads for a single hydrate.

2. **ClockSweeper threads are live throughout** (76 samples / 2.7 % of
   wall). Two background threads sweeping the record-page and fragment
   caches every 100 ms — most samples are in `ClockSweeper.run` park
   states. Not hot; unlikely to matter.

3. **Graal JIT compiler / PGO** samples appear in the non-Sirix slice
   — `LibGraalClassLoader_*` frames show ~3-5 % of samples. That's JIT
   tiering up the hot kernel path during cold open. Fixed cost of
   JIT warmup. Addressable only via native-image (graal#13377 blocked).

## Concrete iter#04 hypothesis (to validate with targeted profile)

The 2s "DB open + class init" budget is mostly the _tail end of_
projection hydrate (1.2s) + JIT warmup (0.3s) + prefetcher burst
(0.5s of futex wait). Strip out the hydrate and the residual DB open
is likely sub-400 ms — already small.

**Primary iter#04 candidate: bound the virtual-thread prefetcher
concurrency or switch to a bounded prefetch queue.** Spawning ~320
virtual threads for hydrate has a cost even if each is cheap; futex
wait dominates the wall profile of the 2s window (74.75 % of wall
samples = __futex_abstimed_wait_cancelable64).

**Secondary candidate: pathSummary / pathStatistics eager loading.**
No direct evidence in this profile that it's a problem — need a
"hydrate off, projection off" profile to isolate the remaining
DB-open cost. If pathSummary loading shows up there, deferring it is
a clean win.
